# Discord 스프린트 메시지 설계

이 문서는 Discord 스프린트 스레드에서 버튼과 텍스트를 어떻게 주고받을지 정리한 기준 문서다.

말투는 [토스의 8가지 라이팅 원칙](https://toss.tech/article/8-writing-principles-of-toss)을 따른다.

- 길게 설명하지 않는다.
- 지금 중요한 한 가지를 먼저 말한다.
- 애매한 빈말 대신 바로 도움이 되는 말을 한다.
- 사용자가 지금 어떤 감정일지 먼저 생각한다.

## 핵심 원칙

스레드 안의 대화는 두 종류로 나눈다.

1. 같이 결정해야 하는 구간
- Discovery
- Demo Review

이 구간에서는 bot이 그냥 접수하지 않는다.
- 의견을 준다.
- 추천안을 하나 제시한다.
- 다음으로 정해야 할 질문을 좁혀준다.

2. 내가 알아서 진행하는 구간
- Design Pack
- Demo Build
- Technical Freeze
- Implementation
- Merge

이 구간에서는 bot이 쓸데없이 자주 말하지 않는다.
- 지금 무엇을 하는지 짧게 알려준다.
- 중요한 순간에만 다시 말한다.
- 사용자가 묻지 않아도 되는 상태를 만든다.
- 이때 `ACTIVE`는 실제 stage job이 돌고 있다는 뜻이어야 한다.

## 버튼 규칙

버튼은 최소한으로만 쓴다.

Discord에서는 stage code 대신 아래 이름으로 보여준다.

- `1. 방향 정리 [게이트]`
- `2. 흐름 정리`
- `3. 데모 구현`
- `4. 데모 확인 [게이트]`
- `5. 구현 준비`
- `6. 구현`
- `7. 배포 확인 [게이트]`
- `8. main 반영`
- `9. 완료`

### Discovery

- `다음 단계로 진행`
- `파기 후 종료`

### Demo Review

- `다음 단계로 진행`
- `파기 후 종료`

### Preview Review

- `수정 더 하기`
- `완료, main 반영`
- `파기 후 종료`

`수정 더 하기`를 누르면 `5. 구현 준비`로 돌아가서 다시 한 사이클 돈다.
이 loop는 따로 횟수 제한을 두지 않는다.

### 이미 같은 스프린트가 열려 있을 때

- `이어서 하기`
- `새로 시작하기`

`새로 시작하기`를 누르면 기존 스레드는 잠시 멈추고, 새 실행 번호로 다시 시작한다.

### 자동 진행 단계

- `파기 후 종료`

대화는 계속하면 되고, 버튼은 정말 다음 단계로 넘길 때나 여기서 접을 때만 쓴다.

## 단계 전환 요약

단계가 바뀔 때는 바로 이전 단계에서 결정된 내용을 handoff summary로 함께 보여준다.

- 3~6개 bullet로 짧게 정리한다.
- 이미 결정된 내용 위주로 쓴다.
- 남은 쟁점이 있으면 마지막 bullet에 남긴다.
- `/status`에서도 최근 단계 요약을 다시 볼 수 있어야 한다.

자동 단계가 실패해서 `BLOCKED`가 되면:

- `다시 시도`
- `파기 후 종료`

### /status 안에서

thread 안의 `/status`는 작은 컨트롤 패널처럼 쓴다.

- Discovery / Demo에서는 `다음 단계로 진행`
- Preview에서는 `수정 더 하기`, `완료, main 반영`
- `파기 후 종료`

공개 컨트롤 메시지를 못 찾았을 때도 여기서 바로 처리할 수 있다.

## 리액션 규칙

- `👀`: 메시지를 읽고 처리하기 시작했어요.
- `⏳`: 답을 정리하는 중이에요.
- `✅`: 답변이나 반영이 끝났어요.
- `⚠️`: 처리 중 문제가 생겼어요.

짧은 응답은 `typing`만 켠다.
조금 생각이 필요한 응답은 `👀 -> ⏳ -> ✅`로 간다.

## 메시지 규칙

### 1. Discovery

목표:
- 왜 필요한지
- 성공 기준이 뭔지
- 이번에 안 할 게 뭔지

메시지 원칙:
- 질문이면 답을 준다.
- 긴 메모면 요약하고 빈 구멍을 짚는다.
- 무조건 "내용 받았어요"라고 말하지 않는다.
- 목표, 성공 기준, 다른 서비스 패턴, 선택지 비교 같은 PM형 탐색 질문도 받을 수 있어야 한다.
- 이때는 너무 빨리 결론만 내리지 말고, 문제 정의와 선택지 비교를 같이 도와준다.

### 2. Demo Review

목표:
- 이 흐름으로 가도 되는지
- 지금 바꿔야 할 게 뭔지
- 기능 진입점과 완료/취소 위치가 자연스러운지
- 디자인 시스템을 일관되게 사용했는지

메시지 원칙:
- 완성도 평가보다 방향 확인을 먼저 돕는다.
- 진입점, 화면 이동, 완료/취소 상태, 디자인 일관성을 짧게 정리한다.
- 가능하면 스프린트 문서와 핵심 흐름 파일을 첨부해서 바로 다운로드해볼 수 있게 한다.

### 3. 자동 진행 단계

목표:
- 사용자가 계속 관리하지 않아도 되게 만들기

메시지 원칙:
- 지금 내가 하는 일을 먼저 말한다.
- 언제 다시 알릴지 같이 말한다.
- 작은 중간 상태는 숨기고, 의미 있는 마일스톤만 알린다.

## 알림 규칙

자동 진행 단계에서는 아래 순간만 기본 알림 시점으로 본다.

- 데모 확인이 필요할 때
- 데모 URL이 준비됐을 때
- 구현 범위가 갈릴 때
- 검증이 끝났을 때
- merge 전 사람 확인이 필요할 때
- 완료됐을 때

즉 "아직 진행 중이에요"를 반복해서 보내지 않는다.

## 시뮬레이션

### 시뮬레이션 1. Discovery에서 제품 방향 질문

사용자:
```text
캘린더 데이터가 로컬에만 있으면 재설치 때 날아가잖아.
그래서 로그인을 붙이고 싶은데, 처음부터 로그인 강제가 맞을까?
아니면 일단 쓰게 하고 나중에 로그인하게 할까?
```

bot:
```text
내 생각은 이래요.
- 처음부터 로그인 강제보다는, 먼저 쓰게 하고 나중에 로그인하게 하는 쪽이 좋아요.

이유
- 처음 진입 장벽이 낮아요.
- 앱 가치를 느낀 뒤라 로그인 전환이 더 자연스러워요.
- 재설치, 기기 변경, 백업 문제도 같이 풀 수 있어요.

지금 먼저 정할 것
- v1에서 멀티디바이스 동기화를 약속할지
- 로그인 직후 로컬 데이터와 서버 데이터를 어떻게 합칠지
```

### 시뮬레이션 2. Discovery에서 긴 메모

사용자:
```text
이번엔 구글 로그인만 먼저 하고 싶고,
애플 로그인은 iOS에서만 우선 고려하고 싶어.
이메일 가입은 아예 이번 범위에서 빼고 싶어.
```

bot:
```text
이렇게 이해했어요.
- 웹은 구글 로그인 우선
- iOS는 애플 로그인을 열어둘 수 있음
- 이메일 가입은 이번 범위에서 제외

지금 먼저 정하면 좋은 것
- 이번 스프린트의 성공 기준 1~2개
- 이번에 꼭 풀어야 하는 사용자 문제 한 줄
```

### 시뮬레이션 3. Discovery 게이트 통과 직후

bot:
```text
좋아요. 이제 **핵심 흐름 정리** 단계로 넘어가요.

내가 지금 할 일
- 핵심 경로만 좁혀서 정리하기
- 데모에 꼭 필요한 화면만 남기기
- 다시 데모 확인 단계로 가져오기

다시 알려드릴 때
- 데모를 확인해달라고 요청할 때
- 범위를 줄여야 할 때
- 방향이 갈릴 때
```

### 시뮬레이션 4. Demo Review 피드백

사용자:
```text
첫 화면 문구가 너무 길어.
구글 로그인 버튼이 더 위에 있었으면 좋겠어.
```

bot:
```text
데모 피드백으로 이렇게 정리할게요.
- 첫 화면 문구는 더 짧게
- 구글 로그인 버튼은 더 빨리 보이게 위로

이번 단계에선 이 두 가지만 보면 돼요.
- 이 흐름으로 가도 되는지
- 지금 바꿔야 할 게 뭔지
```

### 시뮬레이션 5. 구현 단계에서 진행 안내

bot:
```text
지금은 **구현** 단계예요.
- 구현, 검증, 리뷰를 병렬로 진행해요.

이 구간은 내가 알아서 진행할게요.
중간에 계속 말 걸지 않아도 괜찮아요.
아래처럼 중요한 순간에만 다시 알려드릴게요.
- 막히는 결정이 생길 때
- 검증이 끝났을 때
- 사람이 확인해야 할 게 생길 때
```

### 시뮬레이션 6. 구현 단계에서 blocker 발생

bot:
```text
여기서 결정 하나만 필요해요.

지금 갈리는 점
- 기존 로컬 데이터와 서버 데이터를 병합할지
- 로그인 시 서버 데이터를 우선으로 덮어쓸지

내 추천
- 첫 로그인 때는 로컬 데이터를 1회 업로드하고 서버를 원본으로 바꾸는 쪽이 더 단순해요.
```

### 시뮬레이션 7. 완료 알림

bot:
```text
여기까지 끝났어요.

이번에 끝난 것
- 구현
- 검증
- 반영

다음에 보면 좋은 것
- 실제 사용자 온보딩 이탈이 줄었는지
- 로그인 전환이 자연스러운지
```

### 시뮬레이션 8. 같은 스프린트를 다시 시작할 때

bot:
```text
같은 스프린트가 이미 열려 있어요.
- 스레드: #sprint1-oauth
- 실행: 기본 실행
- 단계: 방향 정리
- 상태: 사람 결정 대기

어떻게 할까요?
```

버튼:
- `이어서 하기`
- `새로 시작하기`

`새로 시작하기`를 누른 뒤:

bot:
```text
새 스레드로 다시 시작했어요.
- 새 스레드: #sprint1-oauth-2
- 실행: 2차 실행
- 이전 스레드는 잠시 멈춰둘게요: #sprint1-oauth
```

### 시뮬레이션 9. 스프린트를 여기서 접을 때

사용자:
- `파기 후 종료` 버튼 클릭

bot:
```text
정말 여기서 끝낼까요?
- worktree를 지우고
- 이 스레드를 종료 상태로 바꾸고
- 스레드를 닫아둘게요.
```

버튼:
- `정말 종료할게요`
- `계속 진행할게요`

확정 뒤:

bot:
```text
이번 스프린트는 여기서 끝낼게요.
- worktree는 정리했어요.
- 이 스레드는 종료 기록으로 남기고 닫아둘게요.
```

## 하지 않을 것

- 모든 메시지에 답장하기
- 질문에도 "내용 받았어요"만 말하기
- 진행 중 메시지를 자주 보내서 피로하게 만들기
- 한 메시지 안에 여러 결정을 한꺼번에 몰아넣기
