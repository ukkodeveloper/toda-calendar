# sprint1-oauth

## Discovery

### Service History And Policy Notes

- Toda의 OAuth는 가입 기능보다 백업, 복구, 멀티디바이스 연속 사용을 위한 장치로 정의한다.
- 첫 진입 강제 로그인은 피한다.
- 사용자가 캘린더 가치를 느낀 뒤 로그인 가치를 보여주는 유도형 흐름을 택한다.
- 서비스 데이터의 기준 식별자는 OAuth provider가 아니라 Toda `user id`다.
- `User`와 `AuthIdentity`를 분리해 provider 확장에 대비한다.
- 동일 이메일 기반 자동 계정 통합은 v1에서 하지 않는다.

### Current Problem

- 재설치나 기기 변경 뒤 기존 캘린더를 안전하게 복구할 수 있는 장치가 없다.
- 멀티디바이스에서 같은 캘린더를 자연스럽게 이어 쓰는 기준 계정 구조가 없다.
- 초기에 로그인 장벽을 높이지 않으면서도 백업, 동기화 가치를 명확히 보여줄 설계가 필요하다.
- 이후 Google, Apple 외 provider가 늘어나도 흔들리지 않는 auth 모델이 필요하다.

### Why Now

- 캘린더는 사용 데이터가 쌓일수록 보존 가치가 커진다.
- 서버 원본과 계정 구조를 늦게 도입하면 이후 복구, 동기화, provider 확장 비용이 커진다.
- 이번 스프린트에서 v1 원칙을 고정해야 데모 리뷰와 기술 동결을 짧게 가져갈 수 있다.

## PRD Lite

### Goal

- 비로그인으로 시작한 사용자가 데이터 보존 필요를 느끼는 시점에 OAuth를 연결하고, 재설치와 멀티디바이스에서도 같은 캘린더를 안전하게 이어 쓰게 만든다.

### Success Criteria

- 로그인 유도 노출 대비 로그인 완료율을 추적할 수 있다.
- 로그인 완료 사용자 중 첫 백업 또는 첫 동기화 완료율을 추적할 수 있다.
- 재설치 또는 새 기기 로그인 뒤 복구 성공 여부를 확인할 수 있다.
- 로그인 도입 이후에도 첫 사용 흐름 이탈이 유의미하게 악화되지 않는다.

### Non-Goals

- 첫 진입 강제 로그인
- 동일 이메일 자동 계정 통합
- 복수 provider 연결 UX
- 정교한 양방향 자동 충돌 해결
- 무거운 분석 스택 도입

### Scope

- OAuth provider는 Google, Apple을 v1 후보로 둔다.
- 로그인 유도는 첫 일정 1개 작성 직후 1차로 노출한다.
- 설정 진입과 백업 또는 동기화 액션 직전에는 강한 유도를 추가한다.
- 로그인 후 데이터 원본은 서버다. 로컬은 캐시와 오프라인 대응 용도로 둔다.
- 첫 로그인 시 현재 기기 로컬 데이터를 서버 계정의 시작 데이터로 업로드하는 기본 경로를 둔다.
- 기존 서버 데이터가 있는 계정으로 로그인할 때는 서버 우선 복구를 기본으로 한다.
- 설정 페이지에서 현재 기기 로그아웃을 지원한다.
- 세션 정책은 사용자 체감 기준 `1년 유지`를 목표로 하고, 내부 토큰은 더 짧게 운용한다.
- 핵심 퍼널 이벤트만 최소 비용으로 적재한다.

### Launch Or Marketing Notes

- 메시지 중심은 가입이 아니라 `캘린더를 잃지 않기`, `새 기기에서도 이어 쓰기`다.
- 첫 화면 카피보다 설정, 백업, 동기화 진입 지점의 가치 설명이 더 중요하다.

## UX Decisions

### Ambiguous Core Paths To Design

- 언제 로그인 가치를 보여줄지
- 첫 로그인에서 로컬 데이터를 어떻게 안전하게 서버 계정 시작점으로 올릴지
- 이미 서버 데이터가 있는 계정으로 로그인했을 때 로컬과 서버를 어떻게 안내할지
- 세션이 길게 유지되는 서비스에서 언제 다시 인증을 요구할지

### Key UX Decisions

- 핵심 경로는 5개 화면으로 좁힌다.
- 화면 1: 비로그인 홈
  - 로그인 없이 바로 캘린더를 쓴다.
  - 상단이나 첫 진입에서는 로그인 압박을 주지 않는다.
  - 홈 우측 상단에 톱니바퀴 아이콘으로 설정 진입점을 둔다.
- 화면 2: 백업 유도 시트
  - 트리거는 `첫 일정 1개 작성 직후`, `설정 진입`, `동기화/백업 액션 직전`으로 둔다.
  - 가치 문구는 `잃지 않기`, `새 기기에서 이어 쓰기`에만 집중한다.
  - 선택지는 `Google로 계속`, `Apple로 계속`, `나중에` 세 개만 둔다.
- 화면 3: 첫 백업 확인
  - 현재 기기 데이터를 계정 시작 데이터로 올린다는 점을 짧게 확인한다.
  - 절대 조용히 덮어쓰지 않는다.
  - 결과는 `백업 완료`와 `이제 다른 기기에서도 복구 가능` 두 메시지로 끝낸다.
- 화면 4: 기존 계정 복구 선택
  - 서버 데이터가 이미 있으면 서버 데이터를 기준으로 가져온다.
  - 로컬에 미동기 데이터가 남아 있으면 바로 병합하지 않는다.
  - v1 데모에서는 `서버 데이터로 복구`를 기본 CTA로 두고, 로컬 보존 또는 비교는 후속 과제로 미룬다.
- 화면 5: 설정 페이지
  - 설정은 계정 중심 범위만 우선 둔다.
  - 최하단에 `로그아웃` 버튼을 둔다.
  - 로그아웃 확인 팝업 제목은 `로그아웃하시겠어요?`로 둔다.
  - 팝업 설명은 `이 기기에서 로그인 상태가 해제돼요. 저장된 일정은 그대로 볼 수 있어요.`로 둔다.
  - 확인 버튼은 `로그아웃`, 보조 버튼은 `취소`로 둔다.
  - 로그아웃은 현재 기기 세션만 해제한다.
  - 로그아웃 후에도 로컬 일정과 캐시는 남기고 서버 동기화만 중단한다.
- 세션 경험 원칙
  - 평소에는 로그인 상태가 1년 유지되는 것처럼 보이게 한다.
  - 내부 access token, refresh token, device session은 분리해 운용한다.
  - 민감한 행위나 이상 징후에서만 재인증을 건다.
  - v1 데모에는 세션 만료 UX를 넣지 않는다.
- 이벤트 로깅 원칙
  - `로그인 유도 노출`, `로그인 시작`, `로그인 성공`, `로그인 실패`, `첫 백업 완료`, `복구 성공`만 우선 수집한다.

  - 이벤트 적재 지점 정의
- `apps/mobile`
  - 이번 스프린트에서는 화면 구현 범위에 넣지 않는다.
  - 다만 동일 정책을 재사용할 수 있도록 문서 기준만 맞춘다.
- `packages/contracts`
  - API가 추가되면 `세션 상태`, `로그아웃 결과`, `백업/복구 상태` 계약만 최소 범위로 정의한다.
  - 이번 단계에서는 불필요한 선확장 계약 추가를 금지한다.

### Shared Contracts

- 서비스 데이터의 기준 주체는 Toda `user id`다.
- OAuth provider 계정은 `AuthIdentity`로 분리 관리한다.
- 동일 이메일 자동 계정 통합은 v1에서 하지 않는다.
- 로그아웃은 `현재 기기 세션만 해제`를 기본 계약으로 둔다.
- 로그아웃 후 로컬 데이터는 남고 서버 자동 동기화만 중단된다.
- 사용자 체감 로그인 유지는 `1년`으로 두되, 내부 토큰과 세션 모델은 더 짧은 주기로 회전 가능해야 한다.
- 서버 원본 계정으로 로그인했을 때 서버 데이터가 있으면 `서버 우선 복구`를 기본 계약으로 둔다.
- 로컬 미동기 데이터의 병합 정책은 v1에서 자동 정교 병합보다 `안전 우선`을 택한다.

### Dependency Order

- 1. `web`에서 설정 진입점, 설정 화면, 로그아웃 UX, 로그인 유도 트리거 UI를 먼저 고정한다.
- 2. `api`에서 현재 기기 로그아웃, 세션 갱신, 백업/복구 기본 경로 계약을 맞춘다.
- 3. `contracts`가 필요하면 마지막에 최소 필드만 추가한다.
- 4. `mobile`은 이번 구현 범위에서 제외하되, 후속 이식 시 동일 문구와 정책을 따르게 한다.
- 5. 백엔드 세션 세부안이 정리되기 전까지 토큰 수명 숫자는 프론트에 하드코딩하지 않는다.

### Merge Guardrails

- auth/session 모델은 `docs/ai/escalation-policy.md`의 상시 확인 대상이므로, 구현 중 세부 정책이 바뀌면 사용자 확인 없이 범위를 넓히지 않는다.
- 현재 스프린트에서는 `설정 UI`, `현재 기기 로그아웃`, `첫 일정 작성 후 로그인 유도`, `서버 원본 복구 기본 경로` 밖의 확장을 금지한다.

- filesystem 기반 임시 저장이나 캐시 정책이 배포 환경에서 유지되지 않을 수 있으면 `Delivery Notes`에 명시한다.
- `P1`, `P2` 리뷰 이슈 또는 필수 검증 실패가 남아 있으면 자동 머지를 막는다.
